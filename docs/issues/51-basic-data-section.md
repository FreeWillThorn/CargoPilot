## Problem Statement

CargoPilot currently has master-data editing spread across several places. Suppliers, customers/consignees, warehouses, and company/system information appear in 管理/设置, 订单详情, 仓库盘点, standalone master-data pages, or workflow-specific drawers. This makes the app harder to navigate and causes ownership conflicts: a workflow page should let users choose or view master data, but it should not become the editing home for every shared object.

The user wants a new first-class section named 基础资料. This section should be the single Admin-only place to create, edit, delete, and maintain Supplier, Customer/Consignee, Warehouse, and Company/System profile information. Existing scattered entry points for editing these objects should be removed.

## Solution

Add a new Admin-only left-navigation section named 基础资料. The section contains four master-data blocks or tabs:

- 供应商
- 客户
- 仓库
- 公司信息

基础资料 owns all create/edit/delete forms for these objects. Workflow sections still display and select master data where needed, but their add/edit/delete buttons for these shared objects are removed.

订单详情 keeps customer selectors and customer display, but no longer owns 收货客户 creation/edit/deletion. 仓库盘点 keeps the selected warehouse context and warehouse information display, but no longer owns 仓库资料 creation/edit/deletion. The top-right 管理/设置 menu should not expose Supplier, Customer, Warehouse, or Company/System profile editing after this migration.

## Implementation Decisions

- Add 基础资料 as an Admin-only left-navigation section.
- Keep Warehouse User navigation unchanged: Dashboard, 订单详情, 货物详情, 仓库盘点.
- 基础资料 owns Supplier CRUD, Customer/Consignee CRUD, Warehouse CRUD, and Company/System profile editing.
- Use existing master-data helpers and existing system setting storage where possible.
- Reuse existing Supplier, Consignee, Warehouse, and Settings forms where possible, but render them under 基础资料.
- The page can be one route with tabs/sections for 供应商, 客户, 仓库, 公司信息. Do not create four unrelated top-level nav sections.
- Remove or hide the top-right 管理/设置 menu as the primary entry for these records.
- Remove 收货客户 create/edit/delete panel from 订单详情. Keep order forms/selectors that choose existing customers.
- Remove 仓库资料 create/edit/delete controls from 仓库盘点. Keep warehouse selectors and warehouse information display.
- Remove standalone master-data pages from primary navigation. Legacy URLs may redirect to 基础资料 or continue as compatibility endpoints, but they must not be visible as duplicate entry points.
- Company information includes seller company name, address, phone, email, tax/business registration number, bank information, default origin country/port, default currencies, reminder lead days, and container reference limits.
- Preserve role checks: Admin User can access 基础资料; Warehouse User cannot access it or its post endpoints.
- Preserve audit/logging behavior where existing create/edit/delete helpers already record changes. If some helper lacks logging, do not invent broad new audit infrastructure in this issue.
- Keep delete behavior conservative: if a record is referenced by orders, goods, receiving, or other data, show a clear error instead of deleting.
- Keep field labels Chinese-first and business-friendly.

## Out of Scope

- Customer login portals.
- Supplier portals.
- Multi-address customer records.
- Multi-contact supplier/customer records.
- Full organization/account-management features.
- Supplier scoring, supplier payment status, customer credit, tax invoice workflows, or accounting ledger behavior.
- Redesigning all visual styling beyond fitting 基础资料 into the existing drawer/modal/table patterns.
- Changing how workflow selectors reference master data.

## Further Notes

- This PRD supersedes the earlier UI ownership decision that placed 收货客户 CRUD in 订单详情 and 仓库资料 CRUD in 仓库盘点.
- Workflow sections still use master data. They simply stop owning master-data CRUD.
- The simplest implementation should reuse existing forms and handlers first, then consolidate UI entry points. Do not build a new abstraction layer unless existing helpers cannot support the new section.
