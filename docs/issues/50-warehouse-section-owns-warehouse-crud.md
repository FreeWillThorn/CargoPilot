**Summary:** Move Warehouse master-data CRUD into 仓库盘点 and clean up 管理/设置 ownership.

**Current behavior:**

仓库盘点 lets users choose a warehouse and record receiving activity. Warehouse master data is managed separately under 管理/设置. This splits a workflow object from the section where it is used.

**Desired behavior:**

仓库盘点 should own Warehouse master-data management:

- create warehouse
- edit selected warehouse
- delete warehouse when safe
- select warehouse and view inventory/receiving records

Actions should use compact buttons and right-side drawers/modals consistent with the rest of the app. Admin Users can manage warehouses. Warehouse Users can view warehouse details and record permitted receiving activity, but cannot create/edit/delete warehouse master data.

After create/edit/delete, remain in 仓库盘点 and preserve a sensible warehouse context. If deletion is blocked by related data, show a clear error rather than crashing.

Top-right 管理/设置 should no longer be the primary place for 仓库资料 after migration. It may keep system settings and any master data not yet embedded in workflow sections.

**Key interfaces:**

- Warehouse list/context selector in 仓库盘点.
- Existing create/update warehouse helpers.
- New delete warehouse behavior if not already present.
- Utility menu rendering and route permissions.

**Acceptance criteria:**

- [ ] 仓库盘点 exposes Admin-only create/edit/delete warehouse actions.
- [ ] Warehouse create/edit uses drawer forms with Chinese labels.
- [ ] Warehouse delete uses a simple confirmation and handles related-data failures gracefully.
- [ ] Warehouse User cannot create/edit/delete warehouses.
- [ ] 管理/设置 no longer presents 仓库资料 as the primary workflow entry after migration.
- [ ] Tests cover create/edit/delete action visibility, Admin flow, and Warehouse restriction.
